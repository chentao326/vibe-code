# 任务描述 Pattern 沉淀

> rubric.md 教 AI **怎么打分**，本文件教 AI **怎么写任务描述**。两者解耦。
> 所有 pattern 必须可追溯到具体任务 + 复盘数据。

---

## 核心 meta 原则

Pattern 是 toolbox 不是 mold。每个任务属性不同，先问"这个任务最强属性是什么"，再选 pattern。

---

## 结构选型 cheat sheet

| 任务属性 | 适合的 task 结构 |
|---|---|
| 修 bug | 现象 → 根因假设 → 修复方案 → 验证步骤 |
| 加 feature | 用户故事 → 具体需求 → 验收标准 → 约束 |
| 重构 | 现状问题 → 目标结构 → 迁移步骤 → 风险 |
| 技术调研 | 问题 → 候选方案 → 评估维度 → 结论 |
| 依赖升级 | 当前版本 → 目标版本 → breaking changes → 测试计划 |

---

## 有效 task 描述的特征（从复盘反推）

### Pattern 1: 精确到文件路径 + 函数名
❌ "修复登录bug"
✅ "src/auth/token.ts 的 refreshToken() 在 token 过期时返回 401"

### Pattern 2: 验收标准可操作
❌ "确保正常"
✅ "登录后 30 分钟内不出现 401；auth 测试全部通过"

### Pattern 3: 约束前置
❌ 做到一半 AI 改了 API 签名 → 其他模块崩溃
✅ "不修改 API 接口签名（其他模块依赖）"

### Pattern 4: 上下文充分但不过度
- 关联文件 2-5 个：列出
- 关联文件 >10 个：描述模块名即可
