**最后更新**: 2026-05-24

## v0 综合分公式（等权起步）

```
composite = (CS×1.0 + CX×1.0 + AM×1.0 + TE×1.0 + AQ×1.0) / 5 × 2.0
```

每个维度 0-5 整数分。综合分范围 0-10。

---

## 5 个维度

### CS — Clarity of Spec（需求清晰度）
*任务描述有多清晰、具体、可执行？*

- **0** — "帮我改个bug"（完全无法执行）
- **3** — 描述了问题现象和期望结果，但缺少具体文件/行号
- **5** — 精确到文件路径+函数名+期望行为+验收标准

### CX — Cross-cutting Impact（改动影响面）
*这个任务涉及多少模块/文件/系统？*

- **0** — 单文件，纯函数，无外部依赖
- **3** — 同模块内 3-5 个文件
- **5** — 跨 5+ 模块，涉及数据库 schema + API + 前端

### AM — Ambiguity（隐性知识占比）
*多少信息在脑子里没写进任务描述？*

- **0** — 所有信息已写在 task 描述中
- **3** — 需要一些业务背景知识，但可推断
- **5** — 大量业务逻辑和设计决策只在用户脑子里

### TE — Testability（可验证性）
*多容易判断"做对了"？*

- **0** — 结果纯主观，无法自动验证
- **3** — 有明确的验收标准但无自动测试
- **5** — 有完整的自动化测试覆盖 + 编译检查

### AQ — Agent Quality Match（AI 匹配度）
*这个任务是否适合 AI 执行？*

- **0** — 需要 AI 工具链不支持的能力（如 GUI 操作）
- **3** — AI 能做但需要多轮澄清
- **5** — AI 原生强项：代码生成、重构、bug 修复

## 待验证观察

### 2026-05-24 修复加固vibe-code — TE维度价值验证，macOS隐式行为是主要风险因子

- **任务**: 修复并加固 vibe-code（5 文件修改 + 4 测试文件新建）
- **综合分**: 8.0（CS=5, CX=3, AM=2, TE=5, AQ=5）
- **实际结果**: ~10 轮工具调用、~25min、1 bug（awk `(+)` 后缀）、3 测试因 pipefail 失败后修复
- **判断**:
  - TE=5 的测试覆盖在首次运行中暴露了 awk 匹配 bug——验证了可测试性维度对软件工程的价值
  - macOS BSD 工具链差异是 AI 编码的真实风险因子：awk 字段格式、date 参数、pipefail 行为
  - AM=2 的判断被验证：Bash 的隐式行为（set -e + pipefail）需要领域经验，不是纯推理能覆盖
  - CX=3 的"模块跨度"影响小于预期——每个模块的改动局部性高时，跨模块不增加复杂度
- **Rubric 调整建议**:
  - TE 在软件工程领域权重可能应该 > ×1.0（测试驱动 bug 发现的杠杆效应 > 内容领域）
  - AM 在 shell 脚本场景可能需要细分——Bash 的隐式行为密度远高于应用代码
  - CX 可能应该拆分为"文件数"和"模块数"两个子维度
- **详见**: predictions/2026-05-24_8a3f2c71e901_fix-and-harden-vibe-code.md

### 2026-05-24 Git初始化+装hook — 首个盲预估样本：轮次低估25%，验证环节成本被忽略

- **任务**: Git 初始化 + 安装 hook（8 步 CLI 操作）
- **综合分**: 7.6（CS=5, CX=3, AM=2, TE=4, AQ=5）
- **预估**: 5-8 轮、10-15min、🟢 低风险、9/10 满意度
- **实际**: ~10 轮、~10min、1 bug（hook 配置格式）、8/10 满意度
- **偏差**: 轮次 ❌ 低估 25%，耗时 ✅ 命中，风险 ✅ 命中，满意度 ❌ 略高估
- **判断**:
  - 低估主要来自验证环节（hook 测试编辑 + 回退 + 确认 = 3 轮）——预估时只算了"做"的轮次，没算"验"的轮次
  - 关键假设 3（hook 通过 stdin 通信）被推翻——Claude Code 用 env vars + command args
  - CS=5 仍有 AM 盲区：框架 API 细节（settings schema）无法通过更好的任务描述消除
  - 耗时准确说明 CX=3 的文件跨度估计合理
- **Rubric 调整建议**:
  - TE 维度需区分"自动化验证"和"手动验证"——手动验证有隐性交互成本
  - 预估轮次时应显式拆分"做 N 轮 + 验 M 轮"，而不是合并估计
  - AM 的"框架/平台隐式 API"子项值得独立追踪
- **详见**: predictions/2026-05-24_3f7a1b_install-hooks-init-git.md

### 2026-05-24 创建CLAUDE.md — 首个全中盲预估，CX=1任务精度显著高于CX≥3

- **任务**: 创建项目 CLAUDE.md（单文件，94 行）
- **综合分**: 6.8（CS=5, CX=1, AM=2, TE=4, AQ=5）

- **实际**: 3 轮、~5min、0 bug、9/10 满意度
- **偏差**: 全部命中 ✅（四个指标 + 四条假设全部验证）
- **判断**:

  - TE=4 的验收标准（wc -l、grep emoji）再次发挥了即时反馈价值

- **Rubric 调整建议**:
  - CX 可能是预估精度的主导预测因子——值得在 ≥5 样本后做相关性分析
  - 低 CX 任务（1-2）的预估范围可以比高 CX 任务（3-5）收得更窄
- **详见**: predictions/2026-05-24_a1b2c3_add-claude-md.md

### 2026-05-24 测试install.sh — CX=1连续全中，Bash参数歧义已成规律

- **任务**: 为 install.sh 编写测试（单文件，20 个测试用例）
- **综合分**: 6.8（CS=5, CX=1, AM=2, TE=4, AQ=5）
- **预估**: 3-5 轮、5-10min、🟡 中风险、8/10 满意度
- **实际**: 4 轮、~5min、1 bug（grep `--` 前缀）、9/10 满意度
- **偏差**: 全部命中 ✅，满意度超预期（9 > 8）
- **判断**:
  - CX=1 连续两个任务全部指标命中——规律支持 CX 是预估精度的主导因子
  - grep `--` bug 和 task 1 的 pipefail 同根：Bash CLI 参数歧义——应成为 AM 评分固定检查项
  - TE=4 的杠杆效应：AI 产出 20 个测试（远超 8 个最低要求），高 TE 天然提升交付质量
  - 偏差方向从第 1 个任务的"低估"转向"全中"——校准正在生效
- **Rubric 调整建议**:
  - AM 在 Bash 任务中可自动 +1（Bash 的隐式行为密度是稳定风险因子）
  - CX 对预估精度的预测力值得在 bump 时做定量分析
- **详见**: predictions/2026-05-24_d4e5f6_test-install-sh.md
