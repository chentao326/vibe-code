# Rubric Bump Validation Protocol（升级验证协议）

被这些子 skill 引用：`vibe-bump`、主 SKILL.md。

这是项目原则 #2 的完整规范——**升级 = 全量重打**。

---

## "升级"的定义

下列任一变更触发完整 bump 流程：

- 维度系数变化（CS×1.0 → CS×1.5）
- 维度增减（新增/删除维度）
- 维度定义颠覆性改写
- 归一化常数变化

下列变更**不**触发 bump，但要在 `rubric.md` 标注：
- 维度定义边际细化
- 锚点样本更新

---

## 何时可以提议升级

任一成立即可提议——但**最终是 AI 判断观察的成熟度 + 信号强度**：

1. 某个跨任务观察出现清晰可追溯的支持模式（默认参考 ≥3 任务）
2. 当前 rubric 在最近复盘中**系统性偏向同一方向**（默认参考 ≥3 次同向）
3. 某候选维度展示了独立预测力
4. 校准池跨过某分水岭（5/10/20/50）触发常规复审

**AI 提议 bump 时必须显式标注**：`default-aligned`（满足常规门槛）还是 `judgment-driven`（基于强信号但样本少）。

---

## 何时禁止提议升级（硬约束）

- 校准池 < 5 样本时一般不 bump（除非出现强反例如 ≥3x 偏差）
- 上次 bump 后无新校准样本——必须至少 1 个新样本

---

## 完整升级流程（强制 5 步，不允许跳步）

### Step 1：写出新公式的完整方程

不能只说"CS 权重提高"，必须把所有系数和归一化常数完整写出。

示例（合规）：
```
v0: composite = (CS×1.0 + CX×1.0 + AM×1.0 + TE×1.0 + AQ×1.0) / 5 × 2.0
v1: composite = (CS×1.5 + CX×1.0 + AM×1.0 + TE×1.5 + AQ×1.0) / 6 × 2.0
```

### Step 2：校准池全量重打分

校准池定义：`predictions/*.md` 中所有有完整复盘段的文件。

每篇用新公式重算 composite。**不重新打各维度分数**——维度分不变，只重算综合分。

例外：新增维度时必须回追打新维度的分。

### Step 3：计算排序一致性

新 composite 排序 vs 各任务实际表现排序，必须满足：
- 在 ≥`THRESHOLD` 比例任务上一致（默认 4/5 = 80%）
- 不能把旧公式做对的任一对任务顺序搞反

**实际表现排序**：按"效率"排序。效率 = 1/(耗时×迭代次数×bug数+1)。对于 vibe coding，"表现好" = 快速、少迭代、少 bug。

### Step 4：跨模型独立审核（强制）

`CROSS_MODEL_AUDIT=true`（默认）时：
1. 调外部 LLM，打包：旧公式 + 新公式 + 校准池数据 + 排序对照表
2. 外部 LLM 输出 **PASS** / **REJECT** + ≥100 字理由
3. 本地 + 外部都通过才能进入 Step 5

### Step 5：落地 + cleanup

1. 更新 `rubric.md` 顶部版本号 + 公式
2. 写入"升级 Memo"段（触发原因 + 证据 + 新公式 + 已知局限）
3. 删除被吸收/推翻的观察
4. 未解决的观察 → 迁移到"待验证假设"段
5. 所有校准样本 prediction 文件底部追加 `**Re-scored under vN on YYYY-MM-DD**`

---

## bump 被拒后的处理

| 失败位置 | 处理 |
|---|---|
| Step 3 排序不一致 | 候选公式回到"待验证"区。**不允许**放宽 THRESHOLD |
| Step 4 外部审核 REJECT | 完整记录外部理由到 rubric.md |
| Step 4 双方冲突 | 视为 REJECT。不稳定 |

---

## 升级阻尼

校准池每多一个任务，bump 成本线性上升——这是**故意的**。频繁 bump = 在追噪声。稳定 rubric 的特征是 bump 越来越罕见、单次 bump 越来越有分量。
