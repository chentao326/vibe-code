# Observation Lifecycle（观察生命周期协议）

被这些子 skill 引用：`vibe-retro`、`vibe-bump`、主 SKILL.md。

这是项目原则 #3 的完整规范——**rubric 是工作台，不是博物馆**。

---

## 三个生命阶段

```
[新增] → [观察记录] → [跨任务观察] → [规律沉淀] / [被吸收为维度] / [被推翻]
```

| 阶段 | 在 rubric.md 的位置 | 触发 |
|---|---|---|
| **观察记录** | `## 待验证观察` 段 | 单次复盘后写入 |
| **跨任务观察** | `## 跨任务观察` 段 | 同一个 pattern 在 ≥2 个任务里出现 |
| **规律沉淀** | `## 规律沉淀区` 段 | ≥2 任务支持 + 通过升级验证 |

---

## 升级到下一阶段的门槛

| 从 → 到 | 默认门槛 | AI 判断（可软违反） |
|---|---|---|
| 观察记录 → 跨任务观察 | ≥2 任务 | 1 任务 + 极强信号也可升 |
| 跨任务观察 → 规律沉淀 | ≥2 任务 + 不需改公式 | 强信号也可升（标 `Single-task, high-confidence`） |
| 跨任务观察 → 维度 | ≥3 任务 + 通过 bump 流程 | — |
| 任意 → 删除 | 被新数据推翻 / 被吸收为维度 | — |

---

## 删除规则

**两类条目必须删，不能保留**：

### 类型 A：被吸收为维度的观察
被吸收进正式维度 → 删除原观察。维度本身就是新归宿。

### 类型 B：被新数据推翻的观察
新数据证明观察错误 → 删除。保留会让读者困惑"当下规则是什么？"

---

## 必须保留的条目

- 未解决的观察 → 迁移到"待验证假设"段
- 历史校准事件 → 保留在版本日志的"升级 Memo"段
- 仍生效的规律 → 保留在"规律沉淀区"

---

## 不允许的反模式

| 反模式 | 为什么不行 |
|---|---|
| `~~CS 权重 1.0~~` `**改为 1.5**`（带删除线） | 用 git history 看旧值 |
| "我以前觉得 CS 重要但现在发现..." | 考古条目让人不知道当下规则 |
| "保留这条作为反例" | 反例放版本 memo，不放观察段 |
| "下个 bump 再删" | bump 落地的同一次操作里删干净 |

git history 才是真正的归档。`rubric.md` 是当前生效规则的快照——打开它应该看到**今天怎么评估**。

---

## cleanup pass 的强制时机

强制触发：
- bump 落地的最后一步（见 bump-validation-protocol.md Step 5）

非强制但建议：
- 校准池每加 5 个任务，主动审视观察段

---

## 行数预算

| 校准池规模 | 健康行数 | 警戒行数 |
|---|---|---|
| 0-5 任务 | 100-200 | >300 |
| 5-20 任务 | 200-400 | >500 |
| 20+ 任务 | 300-500 | >700 |

行数超警戒 → 必跑 cleanup pass。
