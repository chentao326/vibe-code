# Vibe Code

**把 vibe coding 变成可校准的实验。**

你用 AI 写代码——但你怎么知道自己有没有在进步？每次说"这个任务应该很简单"——准过吗？

Vibe Code 把每一次 AI 编码变成闭环实验：

📊 评估任务 → 🎯 盲预估 → 🚀 开始编码 → 📈 复盘对账 → 🧬 进化你的判断力

跑一个月 = 你的 prompt 能力有数据支撑。
跑三个月 = 你对任务难度/风险/耗时的判断力是三个月前的 10 倍。

---

## 它做什么

**核心闭环**（每次编码都走）：
- **任务前**：AI 评估任务难度、风险、清晰度；写一份盲预估（耗时/轮次/bug 风险）——写完不可改
- **执行中**：自动采集 git diff、lint 结果、耗时数据
- **完成后**：复盘对账——预估 vs 实际，偏差在哪，为什么
- **长期**：评估模型随你的数据进化——不再是通用猜测，是你的专属判断力

**辅助能力**（需要时用）：
- **找选题**：从代码库自动扫描 TODO、测试缺口、过期依赖、高频修改文件——不知道做什么时说一声
- **趋势监控**：安全漏洞、依赖更新、社区动态——确保项目不腐烂
- **代码画像**：分析项目的复杂度热点、测试覆盖分布、依赖健康度——让 rubric 更懂你的项目
- **对标学习**：导入你认可的优质开源项目，反向提取代码质量标准
- **快速打分**：不写文件、不预估，纯 triage——比较多个候选任务时用

## 第一次用

### 新项目（刚创建或还没有什么代码）

在你的项目目录里打开 Codex / Claude Code，说：

```
初始化 vibe-code
```

5 个问题搞定 onboarding。然后按下面的流程走。

### 已有项目（已经写了一段时间）

同样是说 `初始化 vibe-code`，但体验不同——系统会自动检测到你的项目有 git 历史，然后：

1. **告诉你项目状态**："你的项目跑了 180 天，342 个 commit，89 个文件"
2. **问你要不要导入历史任务**：从 git 历史中挑 3-5 个有代表性的 commit（一个 bug 修复、一个 feature、一个重构...），帮你重建任务描述和复盘记录
3. **这些历史任务作为 anchor**：让 rubric 一开始就了解你的项目特征——不用从零冷启动

导入的历史任务标记为 "reconstructed"，**不计入校准池**（因为它们不是盲预估），但能让你的评估模型从一开始就有参考基线。

## 使用流程

核心闭环围绕五个动词：**初始化 → 评估 → 编码 → 复盘 → 升级**。此外还有辅助命令帮你找选题、看趋势、分析项目。

```
                    ┌──────────────────────┐
                    │  vibe-seed  找选题    │  ← 不知道做什么时
                    │  vibe-trends 看趋势   │  ← 关注依赖/安全
                    │  vibe-profile 画像    │  ← 了解项目特征
                    │  vibe-learn-from 对标 │  ← 导入质量标准
                    └──────────────────────┘
                      ↑ 辅助层，需要时调用
                      
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│①init     │→  │②assess   │→  │③ 编码    │→  │④retro    │→  │⑤bump     │
│ 首次设置  │   │ 打分+预估 │   │ AI 干活   │   │ 对账复盘  │   │ 进化公式  │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
                     ↑ 核心闭环，每次编码都走
```

### 第一步：写任务描述

在 `tasks/` 目录下创建一个 `.md` 文件描述你要做的事。

**任务描述应该包含**：
- 目标：一句话
- 上下文：关联的模块/文件
- 具体需求：详细描述
- 验收标准：怎么判断做对了
- 约束：不能改的东西

### 第二步：评估 + 盲预估（编码前必须做）

```
评估这个任务 tasks/2026-05-24_fix-auth-bug.md
```

Agent 会打分、写预估、让你 review。落盘后预估段不可再改。

### 第三步：开始编码

用 AI 正常干活。

### 第四步：复盘对账

```
复盘 tasks/2026-05-24_fix-auth-bug.md
```

自动采集 git diff，对比预估 vs 实际，偏差分析写入 rubric.md。

### 第五步：进化（攒够 5 个复盘后）

```
升级模型
```

强制 5 步验证流程——全量重打 + 排序一致性 + 跨模型审核。

### 辅助命令

| 命令 | 用途 | 场景 |
|---|---|---|
| `打分这篇 tasks/xxx.md` | 快速打分（控制台，不写文件） | 有多个候选，想先 triage 再决定评估哪个 |
| `找选题` | 扫描 TODO/测试缺口/依赖过期/churn | 不知道接下来做什么 |
| `趋势` | 安全公告 + 依赖更新 + 动态 | 定期检查项目健康度 |
| `分析项目` | 代码库画像 | 了解复杂度热点和测试覆盖 |
| `状态` | WIP/待复盘/准确率/提醒 | 任何时候 |
| `推荐任务` | 下一步做什么 | 有多个 task 时 |
| `学这个项目 <url>` | 对标学习 | 校准代码质量标准 |
| `导入历史任务` | 事后补导入 | 初始化时跳过了 |
| `迁移` | 升级 state schema | git pull 后发现新版本 |

## 使用示例

### 示例 1：修一个 token 刷新 bug（核心闭环）

**① 写任务描述**

创建 `tasks/2026-05-24_fix-token-refresh.md`：

```markdown
# 修复登录 token 过期后刷新失败

**创建日期**: 2026-05-24

## 目标
修复 auth 模块 token 刷新：access token 过期后 refresh 请求返回 401。

## 上下文
- 相关文件：src/auth/token.ts, src/api/client.ts
- 用户反馈：登录后约 15 分钟开始报 401

## 具体需求
1. 检查 token.ts 中 refreshToken() 的错误处理
2. 确认 refresh token 请求的 header 正确
3. 修复后：token 过期 → 自动刷新 → 重试原请求

## 验收标准
- 登录后 30 分钟内不出现 401
- 现有 auth 测试全部通过
- 新增：token 过期自动刷新的集成测试

## 约束
- 不修改 API 接口签名
```

**② 评估**

说 `评估这个任务 tasks/2026-05-24_fix-token-refresh.md`

Agent 输出：

```
📊 维度评分
| CS | 5 | 文件路径+期望行为+验收标准都写了 |
| CX | 2 | 仅涉及 auth 模块 2 个文件 |
| AM | 1 | 信息完整 |
| TE | 4 | 有测试可验证 |
| AQ | 5 | 代码修复，AI 强项 |
综合分：6.8 / 10

📊 盲预估
| 预计轮次 | 2-4 轮 |
| 预计耗时 | 10-20 分钟 |
| Bug 风险 | 🟡 中（token 边界情况） |

概率：一轮过 30% | 正常迭代 55% | 多次返工 15%

ok？
```

回 `ok` → 落盘锁定。

**③ 编码**（正常干活）

**④ 复盘**

说 `复盘 tasks/2026-05-24_fix-token-refresh.md`

```
📊 复盘
| 对话轮次 | 预估 2-4 | 实际 3 | ✅ |
| 耗时     | 10-20m  | 14m    | ✅ |
| Bug      | 🟡      | 0      | ✅ 好于预期 |

被验证 ✅：轮次和耗时在预估范围内
新观察 → 写入 rubric.md：CS=5+CX=2 = 低成本任务标准画像
```

## 已有项目的三种使用策略

| 策略 | 怎么做 | calibration 启动速度 | 精度 |
|---|---|---|---|
| **A: 纯盲起步** | 跳过历史导入，所有预估都真正 blind | 慢（5-10 个任务后才能 bump） | 高（无污染） |
| **B: 历史导入（推荐）** | 导入 3-5 个历史任务作为 anchor | 中（3-5 个盲预估后即可 bump） | 中（reconstructed 有回忆偏差但不影响 bump 验证） |
| **C: 激进导入** | 导入 10+ 历史任务 | 快（1-2 个盲预估后即可 bump） | 较低（大量 reconstructed 样本可能让 rubric 过拟合历史模式） |

**推荐 B**：导入 3-5 个代表性任务——够让 rubric 理解你的项目，又不至于让历史回忆偏差主导公式。

## 关键概念

| 概念 | 含义 |
|---|---|
| **Blind prediction（盲预估）** | 编码前写的预估，不可修改——校准价值的根基 |
| **Rubric（评估模型）** | 5 维度 + 权重，v0 等权起步，随数据进化 |
| **Calibration pool（校准池）** | 所有有真正盲预估+复盘的任务——bump 验证数据集 |
| **Reconstructed（重建）** | 历史导入的任务——参考价值，但不进校准池 |
| **Confidence（信心）** | 🔴🟠🟡🟢🔵——如实标注 |
| **Bump（升级）** | 改公式——全量重打 + 排序 ≥80% + 跨模型审核 |
| **WIP（进行中）** | 已评估未完成的任务——积压过多会停止推荐新任务 |

## 为什么不是"又一个项目管理工具"

| Jira/Linear/Notion | Vibe Code |
|---|---|
| 管理"做什么" | 管理"怎么判断" |
| 追踪 deadline | 追踪判断准确率 |
| 给人看的 | 给 AI Agent 看的 |
| 静态配置 | 自动进化的评估模型 |

一句话：那些工具帮你"管任务"。这个工具帮你"管判断"。
